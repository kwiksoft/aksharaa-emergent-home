4. 8. Swapped into the component, removed the old JPEG and its CSS mask-fade workaround (no longer needed — this asset has genuine transparency, so there's no seam to soften).

## Hard-won lessons from this thread (read before doing layout/positioning work)

These are new lessons from this session, **in addition to** the photo-collage lessons in the 2026-06-28 handover (shape category first, continuous grid overlay, circle width=height check, draw-and-compare verification, scale-matched side-by-side, container aspect ratio cascading — all still apply).

1. **`document.scrollWidth` overflow checks do NOT catch clipping caused by `overflow-hidden`.** This was a real bug that shipped silently across at least two commits before being caught. If an element is absolutely positioned inside a container with `overflow-hidden` (very common pattern for decorative background images in this codebase), it can be cut off at one edge while the page itself shows zero horizontal scroll — because the hidden overflow suppresses the very signal the standard check looks for. **Whenever positioning an absolutely-placed decorative element near a breakpoint where its column/container narrows, explicitly measure the element's own bounding box against its containing block's bounding box** (not just the page's scrollWidth) — see the `check_clip.py` / `check_clip_1024.py` pattern used this thread: get both boxes via Playwright, compute `element.x + element.width` vs `container.x + container.width`, confirm the difference is positive (safe) not negative (clipped).
2. **A `max-width` constraint on an element does nothing if the element's actual rendered width is already capped by something else upstream** (its parent column's width, in this case). Before concluding "the cap isn't working," check what's actually constraining the element's rendered size — in a CSS grid, that's frequently the column track width, not anything on the element itself. This cost a wasted round in Section 3 (an explicit `max-w-[564px]` on the image had zero effect until the grid ratio itself was changed).
3. **When a person gives instructions in physical units (cm), convert at 96px/inch (~37.8px/cm) as a web-standard reference point**, and say so explicitly in commit messages / tracker notes — there's no universal "correct" conversion (it depends on assumed viewing distance/DPI), so document the assumption rather than silently picking a number.
4. **Repeated small "move it more" instructions compound risk at breakpoints with little spare room.** Three consecutive "0.5cm right" requests, each individually verified against the *previous* state's constraints, eventually pushed a value past a safety boundary (the lg-breakpoint clipping bug) that none of the individual steps' checks caught, because each check only validated *that step's* delta, not the cumulative position against every constraint from scratch. **Periodically re-verify ALL constraints (gap-to-neighboring-element AND containment-within-parent) from first principles, not just incrementally**, especially after 2+ consecutive small adjustments in the same direction.
5. **A background-removal technique that works well on one image does not necessarily transfer to a visually-similar image from a different export.** The Section 2 desk-props photo and the Section 3 growth-bar illustration both had the exact same "fake transparency" problem (checkerboard baked into JPEG pixels), but the precise per-pixel phase-matching approach that worked perfectly on one **failed** on the other because the second image's checker pattern drifted slightly across the canvas (a different compression/export artifact). The fallback that worked on both eventually was **local-variance thresholding** (checkerboard = low local variance, real content = high local variance) — this is more robust because it doesn't assume a perfectly regular global grid, just a local statistical property. If a future image has the same fake-checkerboard problem, **try local-variance thresholding before phase-matching** — it's the more generally reliable of the two techniques discovered this thread.
6. **`rembg` (u2net) is good at cutting solid, well-defined objects but can mistake a large uniform background region (e.g. a white wall/desk surface) for foreground** if the flattening step beforehand wasn't thorough — it kept a false-positive white wedge shape on the Section 3 image. When this happens, building the alpha mask directly from the variance/phase detection (skipping rembg) was more reliable than rembg on this image — worth trying the direct-mask approach first next time rather than assuming rembg's neural segmentation is always the better fallback.
7. **AI-regenerated "fixes" for a flagged problem (garbled text, fake transparency) can introduce a NEW problem of the same category.** When Tharun's first re-send to fix the fake-transparency issue arrived, it turned out to be a fresh AI regeneration with garbled text ("EMP LOVEES", "PF RSOISTRATION", etc.) — a different defect than the one being fixed. Always re-check a "corrected" asset against the *original* defect categories (text accuracy AND transparency AND general quality), not just the one specific issue that prompted the resend.
