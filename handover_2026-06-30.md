## Important process note — git verification method

This thread used `git push <url> main` followed by `git fetch <url> main` (checking `FETCH_HEAD`) to confirm each push landed on the real GitHub branch. **This method is reliable and was correct every time it was used** — but it does NOT update the locally-cached `origin/main` tracking ref, which only updates via a plain `git fetch origin` or `git pull`. Near the end of this thread, `git status` reported "ahead of origin/main by 21 commits" even though every single one of those commits had already been individually verified as pushed — this was just the stale tracking ref, not a real problem, and was resolved with a plain `git fetch origin`.

**For the next thread:** either keep using the `fetch <url> main` + `FETCH_HEAD` method (it works, just don't trust `git status`'s "ahead/behind" framing afterward — it's reading the stale local ref) or run a plain `git fetch origin` periodically to keep the tracking ref honest. Both are fine; just don't be alarmed by a stale "ahead by N" message if every commit has been individually fetch-verified.

Full layout rebuild per 3 client reference images (a target mockup, a standalone background texture, a cropped illustration). Changed from a centered-heading-above-single-accordion-container layout to a two-column layout: left = kicker/heading/sub-copy + a 3D illustration of a compliance specialist, sitting directly on the page background; right = accordion rebuilt as separate floating white cards (was: one divided container) — open card gets an orange left-border accent + solid minus-circle icon, closed cards get an outline plus-circle icon. Full-bleed decorative background texture behind the whole section.

**The illustration cutout took real effort to get right** — worth knowing about for future asset work: the source image's background was genuinely flat (not a checkerboard fake-transparency trap, confirmed via pixel sampling), but a standard color-distance threshold cutout kept either including a soft ambient glow baked into the image (visible as a halo against test backgrounds) or excluding real subject matter, because the glow's pixel values overlapped the same range as light-toned parts of the actual subject (white folder pages, light laptop screen). Solved by switching to `rembg` (u2net) semantic segmentation instead of color distance — it correctly separates real objects from ambient glow because it reasons about object identity, not raw color. This is probably the right first move for any future "soft glow / vignette" image, rather than starting with color-threshold and discovering this the hard way again.

Column ratio was tuned after catching (via direct bounding-box measurement, not eyeballing) that the heading wrapped to 2 lines at 1440px on the first pass — widened the left column until it fit on one line.

## Key technical lessons from this thread (read before doing similar work)

1. **`overflow-hidden` masks horizontal overflow from `scrollWidth` checks** — this bit the project again this thread (the 2cm-right image move). Any time an absolutely-positioned element is moved within a section that has `overflow-hidden`, check the element's own bounding box against the section's bounding box directly at every relevant breakpoint width, not just the document-level `scrollWidth` comparison.

2. **When "verify the fix" matters, verify with numbers, not just a re-screenshot.** The position-restore correction this thread used a genuinely rigorous method: checked out the known-good commit separately, ran it on a second dev server, and diffed actual rendered bounding-box coordinates between the two. This caught a regression that a visual screenshot comparison alone likely would have looked "close enough" on.

3. **Absolutely-positioned decorative elements that depend on hand-tuned per-breakpoint pixel values are fragile** and accumulate drift over repeated small adjustments (see Section 2's image, which went through 2cm/0.5cm/1cm-up-2cm-left/repositioning rounds before being restructured into normal document flow). When a client reference shows an element as part of a column's natural content stack rather than a floating decoration, restructuring into normal flow is usually more robust than continuing to chase pixel positions — it also tends to fix responsive behavior for free (the old absolute image was invisible below `lg`; the flowed version now appears at every breakpoint).

4. **Translucent card backgrounds (`bg-ak-mist/30` etc.) read fine on plain white sections but look muddy on vivid/non-neutral background images.** This has now come up twice (FAQs, then Documents) — the fix both times was switching to fully opaque white with a drop shadow. Worth checking proactively any time a background image is added to a section that has translucent cards.

5. **Sharing a screenshot description in chat is not the same as sharing a viewable file.** When the client needs to actually see something to decide (the Documents mobile tradeoff), the screenshot needs to be copied to `/mnt/user-data/outputs/` and shared via `present_files` — describing it in words isn't sufficient for the client to make a visual judgment call.

6. **Dev server background processes don't reliably survive across tool calls in this container** — expect to restart with `setsid npm start > logfile 2>&1 < /dev/null &` followed by a ~30s wait periodically throughout a session, same as noted in prior threads.
